## Phase 1 — Fix the "AI is not real AI" problem (Technical Sophistication, 25 pts)

### Task 1.1 — Replace SMA forecasting with a real model
- Remove/retire the simple moving average forecast function.
- Implement forecasting using either: (a) a lightweight Node-side implementation of **Holt-Winters exponential smoothing with trend** (statsmodels-equivalent logic ported to JS, or use the `simple-statistics` / `timeseries-analysis` npm package), or (b) if a Python microservice is acceptable, a small **Facebook Prophet** service called via HTTP from the Express backend.
- Output must include: point forecast **and** a confidence interval (upper/lower bound) for the next N time steps.
- Frontend: update the Recharts forecast chart to render a shaded confidence band (Area component) around the forecast line, not just a single line.
- **Acceptance check:** Forecast API response includes `forecast`, `lower_bound`, `upper_bound` arrays. Chart visibly shows a shaded uncertainty band. README/pitch can now honestly say "time-series forecasting with confidence intervals," not "moving average."

### Task 1.2 — Add an explainability layer to every AI-generated answer
- For every Gemini-generated response (NL query answers, recommendations, anomaly explanations), return and display:
  - The top-k retrieved context chunks/embeddings that were used (from your existing RAG pipeline) — show as an expandable "Sources used" panel.
  - A confidence/relevance score (cosine similarity score from the RAG retrieval step is sufficient — surface the number you already compute internally).
  - One sentence auto-generated by Gemini explaining *why* this recommendation was made, in plain language.
- Build a reusable `<ExplainabilityPanel>` React component; attach it under every AI output card (chat answer, recommendation card, anomaly alert).
- **Acceptance check:** Every AI-generated UI element has a visible "Why this answer?" expandable section with real retrieved data, not placeholder text.

### Task 1.3 — Add a Responsible AI note
- Create `RESPONSIBLE_AI.md` (or a section in README) covering: what data is synthetic vs real, known limitations of the forecasting model, human-in-the-loop requirement for any automated action (see Phase 3), and a one-line bias/fairness disclosure (e.g., forecast quality depends on historical data density per corridor/ward).
- **Acceptance check:** File exists, is linked from README, under 400 words, no fluff.

## Phase 2 — Fix the "single domain" problem (Alignment, 20 pts + Loop, 20 pts)

### Task 2.1 — Add Domain Module: Public Safety / Incident Intelligence
- New backend module (`/backend/modules/publicSafety` or equivalent, matching existing module folder convention) with:
  - A BigQuery table (or local JSON fallback, matching your existing fallback pattern) of incident records for Lucknow: `incident_id, type, ward, lat, lng, timestamp, severity`. Use realistic synthetic data (label it synthetic in the Responsible AI note) if real data isn't available — 200–500 rows across wards/timestamps is enough for a good demo.
  - Reuse the existing anomaly-detection logic (already built for traffic) against this new incident stream — e.g., flag wards with incident-rate spikes vs. their own rolling baseline.
  - A new MapLibre layer (toggleable) plotting incident hotspots, using the same layer-toggle UI pattern as your existing traffic/AQI layers.
- **Acceptance check:** New "Public Safety" toggle in the map sidebar works, anomaly flag appears for at least one synthetic hotspot ward, and it uses the same anomaly code path as mobility (confirm via a shared/imported function, not a duplicated copy).

### Task 2.2 — Add Domain Module: Waste/Environment (ward-level)
- Same pattern as 2.1: `wardId, waste_collection_efficiency_pct, water_quality_index, timestamp`.
- Add one derived metric: a composite "Ward Livability Score" combining AQI (existing) + waste efficiency + incident rate (from 2.1) — this is your strongest cross-domain proof point, build it deliberately.
- **Acceptance check:** A ward detail panel shows all three metrics side by side plus the composite score with a one-line Gemini-generated interpretation of the score.

### Task 2.3 — Cross-domain "Ask the City" query capability
- Extend the existing NL query/RAG pipeline so it can retrieve context from **all three domains** (mobility, public safety, waste/environment) in the same query, not just mobility.
- Add at least 2 canned example queries to the UI (as clickable suggestions) that require cross-domain reasoning, e.g.:
  - *"Which wards have both high AQI and rising incident rates this month?"*
  - *"If we improve waste collection in Ward X, what else might it affect?"*
- The Gemini prompt for these queries must explicitly instruct the model to reason across the retrieved multi-domain context and produce a synthesized answer, not three separate answers concatenated.
- **Acceptance check:** Running the two canned queries returns a single coherent answer referencing data from at least 2 different domain modules, visible in the explainability panel's "sources used."

## Phase 3 — Close the loop: add real automation (Decision-Intelligence Loop, 20 pts)

### Task 3.1 — Automated action-memo generation
- When an anomaly (from either mobility, public safety, or waste module) crosses a defined severity threshold, auto-trigger a backend function that calls Gemini to draft a structured **"Action Memo"**: situation summary, affected ward/corridor, recommended action, urgency level, and a suggested owning department (e.g., Traffic Police, Municipal Corporation, Health Dept).
- Store generated memos in a new table/collection (`action_memos`) with a status field: `draft → approved → dispatched`.
- Build a simple "Action Center" page/tab listing pending memos with an **"Approve & Dispatch"** button (this can be a stub — on click, just flip status to `dispatched` and log a timestamp; you do not need real email/SMS integration for the demo, but say so honestly in the Responsible AI note as a "human-in-the-loop, dispatch integration is a stub for demo purposes").
- **Acceptance check:** Triggering a synthetic anomaly produces a new memo visible in the Action Center within the same session, and clicking Approve changes its status live in the UI.

### Task 3.2 — Citizen reporting → auto-routing (multimodal, doubles as Phase 4 prep)
- Add a simple citizen-facing form: text description + optional photo upload.
- Backend: if photo present, call Gemini Vision (multimodal) to classify the issue type (pothole, garbage, waterlogging, streetlight, other) and extract a short description if the citizen didn't type one.
- Auto-create an entry in the relevant domain module's incident stream with a `source: citizen_report` tag, and plot it on the map immediately.
- **Acceptance check:** Uploading a sample photo of a pothole/garbage pile auto-classifies correctly (or reasonably) and appears as a new map marker within seconds.
